## 第一部分：GenAI 原理：next token

### 这部分要回答的问题

生成式 AI 到底是在“理解”，还是在“预测”？为什么它能写代码、写文章、回答问题，但又经常一本正经地胡说八道？

### 核心讲法

- GenAI 最底层可以先粗暴理解为：`根据已有上下文，预测下一个 token`
- 它不是像人一样“先想明白再说”，而是在不断做条件概率预测
- 之所以看起来像“懂了”，是因为它在海量语料和模式里学到了很强的语言压缩和续写能力
- 之所以会犯错，是因为“像”理解，不等于“真的”理解

### 可以讲到的关键词

- token
- 上下文
- 概率预测
- 采样
- 幻觉

### 这部分听众应该带走什么

AI 很强，但它的强不是“万能智能”，而是“超强的上下文续写和模式生成能力”；理解这一点，后面很多使用方法和局限性都会更好解释。

## 第二部分：chat 模式的三大短板

### 这部分要回答的问题

为什么大家都已经在用 ChatGPT、Kimi、豆包、Claude 这类 chat 工具，但很多时候还是觉得“不稳定”“不够真能干活”？

### 核心讲法

可以把 chat 模式总结成一句话：

`它很适合“我问你答”，但不天然适合“我说你做”。`

### 三大短板

#### 1. 问题闭环

- chat 更像一次性对话
- 它能回答问题，但不天然负责把任务从开始推进到结束
- 缺少目标拆解、过程执行、结果回看这类闭环能力

#### 2. 上下文供给

- chat 里很多问题不是模型不够强，而是你没有把背景交代清楚
- 它不知道你的仓库、团队约束、历史文档、真实目标
- 所以输出经常“看起来对，但不适合你”

#### 3. 知识沉淀

- chat 的交互天然偏一次性
- 这次聊得不错，不代表下次还能直接复用
- 没有把经验沉淀成 prompt、skill、模板、知识库，就容易一直重复劳动

### 这部分听众应该带走什么

不是 chat 没用，而是如果想把 AI 真正接进工作流，只停留在 chat 这一层通常不够。

## 第三部分：Claude Code / Codex 安装与定位

### 这部分要回答的问题

如果 chat 不够，那工程师到底应该开始用什么？Claude Code 和 Codex 这种工具和普通聊天窗口有什么本质区别？

### 核心讲法

- Claude Code / Codex 更接近 `coding agent`，不只是回答问题，而是能进入工作目录、读文件、改文件、执行命令、完成任务
- chat 是“给你建议”，agent 更像“和你一起下场做事”
- 对工程师来说，这种形态的价值明显更高，因为它更接近真实工作流

### 安装部分建议讲法

- 不需要展开讲太细，可以控制在“一句话描述安装”
- 重点不是安装步骤本身，而是让团队知道门槛其实没有想象中高
- 可以顺手讲清楚：
  - Claude Code / Codex 各自依赖什么
  - 哪些场景更适合用它们
  - 和 IDE 内置补全的区别是什么

### 这部分听众应该带走什么

对工程师来说，真正值得开始尝试的，不只是聊天机器人，而是能进入真实工作环境、参与任务执行的 agent 工具。

## 第四部分：prompt 和 skill 的区别

### 这部分要回答的问题

很多人觉得“我会写 prompt”就已经够了，但为什么在工程使用里，skill 这种东西会越来越重要？

### 核心讲法

可以用一个简单对比来讲：

- prompt 更像一次性的指令
- skill 更像可复用的能力封装

### 可以展开的角度

#### prompt 的特点

- 适合临时发问
- 上手快
- 灵活
- 但不稳定，复用性差，容易依赖个人手感

#### skill 的特点

- 适合把一类任务流程固定下来
- 可以带说明、步骤、参考资料、工具约束
- 更容易复用、分享、沉淀到团队内部

### 可以用来辅助解释的比喻

- prompt 像一句临时口头交代
- skill 像一份写好的 SOP + 模板 + 背景资料

### 这部分听众应该带走什么

如果只是自己偶尔试试，prompt 已经够用；但如果希望团队复用、让 AI 真的稳定参与工作流，skill 的价值会越来越大。

## 第五部分：1-2 个现场案例

### 这部分要回答的问题

前面讲了原理、短板、工具、skill，最后一定要落到一个最实际的问题：这些东西到底怎么用？

### 建议案例一：复刻一个网页 / 页面风格

#### 为什么适合演示

- 结果直观，现场反馈强
- 能同时展示 agent 如何读需求、拆任务、改代码、调样式、做验证
- 对工程师来说代入感强

#### 演示重点

- 不是单纯让 AI 输出代码
- 而是展示它如何：
  - 读参考页面
  - 分析结构
  - 在本地项目里修改
  - 运行和验证

#### 听众能看到什么

AI 不只是补全代码，而是真的可以参与一个完整的小任务闭环。

### 建议案例二：写一篇结构化文章 / 分享稿

#### 为什么适合演示

- 对非纯开发同学也友好
- 能展示 AI 不只会写代码，也能参与信息整理、提纲生成、风格改写
- 很适合带出“prompt vs 上下文 vs 沉淀”的差别

#### 演示重点

- 同一个题目，普通 chat 和带足上下文的 agent / 工作流效果有什么差异
- 如何通过补背景、补风格、补标准，让输出越来越像“能用的稿子”

#### 听众能看到什么

AI 在写作、汇报、知识整理上的价值，不是“替你一键写完”，而是“帮你更快进入一个能迭代的初稿状态”。

## 结尾建议

### 核心结论

这场分享最后最好收束到一句简单的话：

`AI 最值得工程师认真投入的，不是多学几个工具，而是开始理解它的工作方式，并把它接进自己的真实工作流。`

### 可以这样收尾

- 先理解一点原理，不用太深
- 再接受 chat 模式的边界
- 再开始用 agent 工具下场做真实任务
- 再把有效经验沉淀成 prompt、skill 或模板

### 希望听众带走的三件事

1. 知道 GenAI 为什么能生成，也为什么会犯错
2. 知道为什么 chat 不够，agent 更接近生产力
3. 知道回去之后可以从一个真实案例开始试

---

## 一句话版总结

这场分享不是为了让大家“知道更多 AI 名词”，而是为了让大家第一次真正明白：GenAI 是怎么工作的，chat 为什么不够，以及工程师怎样把 Claude Code / Codex 这样的 agent 工具接进自己的日常工作。
