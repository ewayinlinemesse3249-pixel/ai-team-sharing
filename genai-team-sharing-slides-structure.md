## 分享目标

这次分享解决一个更实际的问题：团队已经听过很多 AI 工具，也可能零散试过一些，但真正进入工作流的比例并不高。分享不追热点，不做工具盘点，而是用一条主线把原理、工具形态和可复刻案例连起来。

核心判断是：工程师要从 chat 窗口里的问答，转向能进入执行环境的 agent；从一次性 prompt，转向可复用的 skill 和工作流。

## 内容主线

1. GenAI 的基本机制：Next-token Prediction
2. Chat 模式的三类短板：问题闭环、上下文供给、知识沉淀
3. Claude Code / Codex 的定位：AI Coding Agent
4. Prompt 和 skill 的差别：一次性指令 vs 可复用能力
5. 三个现场案例：手绘 UI 到前端页面、复杂项目 spec-driven 工作流、RTM Excel 到 Jira CSV

## 第 9 页：短板一：问题闭环

### 页面标题

短板一：问题闭环

### 核心内容

示例：让 ChatGPT 写一段 Python，你复制到 IDE 里运行，报错后再把错误贴回去，它再改，你再跑。这个过程里，人变成反馈搬运工。

对比：Cursor / Codex / Claude Code 这类工具接入执行环境后，可以写完代码直接跑，看到报错再改，形成 AI 自己驱动的反馈闭环。

### 这一页想讲什么

解释为什么很多人觉得 AI 不可靠：他们一直在用开环 AI。给它执行环境和反馈闭环，可靠性会明显提升。

## 第 18 页：复杂项目，不要让 AI 直接开写

### 页面标题

复杂项目，不要让 AI 直接开写

### 核心内容

参考 `github/spec-kit` 的 spec-driven development 思路，把复杂项目从“直接让 AI 开写”改成：

1. Constitution：先定项目原则
2. Specify：写清 what / why
3. Plan：落技术路线、架构选择和数据边界
4. Tasks：拆成 agent 能逐项完成的任务
5. Analyze：检查规格、计划、任务是否对齐
6. Implement：最后再进入编码、运行、修复和提交

### 这一页想讲什么

复杂项目里，AI 最容易出问题的不是不会写代码，而是太早进入代码。先把意图、约束、计划和任务变成可检查的中间产物，再让 agent 执行。

### 实现备注

已完成。页面主视觉是 `SPEC → PLAN → TASKS → CODE`。

## 第 19 页：案例三：RTM Excel → JIRA CSV

### 页面标题

案例三： RTM Excel → JIRA CSV

### 核心内容

案例目标：读取 RTM Excel 的 Matrix 工作表，生成 Jira 可导入的 CSV。关键不是让 AI 记住字段，而是把规则写成可验收的转换契约。

页面保留五类规则：

1. 输入契约：明确文件类型、工作表名、必读列；用样例 CSV 决定表头和列顺序
2. 字段映射：把 Summary、Issue Type、Priority、自定义字段写成确定规则；未定义字段保持空白
3. 过滤与去重：剔除 uncovered / 空 Summary；按 BR ID + FR ID 去重
4. 输出前校验：表头完全一致、必填字段非空、无 uncovered、无空 Summary、无重复 key
5. skill 化：把一次性 prompt 拆成目标、边界、资源、流程、验收标准和已知陷阱

### 这一页想讲什么

用一个真实的数据转换任务说明：长 prompt 不是终点。它可以被提炼成稳定规则，再沉淀成 skill，让下次同类任务少重来一遍。
